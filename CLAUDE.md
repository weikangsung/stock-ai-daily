# 個股波段 AI 自動化系統

## 系統說明

這是一套針對多隻台股個股設計的波段操作（持有數天到數週）輔助分析系統。

每天早上 8:00 前，系統自動調用 3 個 Sub Agent 進行並行資料蒐集，對觀察清單中的每檔個股進行技術、流動性、消息面綜合分析後輸出波段建議。
收盤後進行檢討，並透過規則驗證機制逐步優化交易規則庫。

### 核心理念
- **自動化**：每日預測與檢討由排程自動執行，無需手動觸發
- **系統化**：每個決策都有明確的依據（技術面、流動性、消息面）
- **可配置**：觀察清單由用戶在 `rules/watchlist.md` 自行定義，可隨時增減標的
- **自我優化**：規則透過持續驗證逐步升級或調整
- **用戶控制**：任何規則變更都需要用戶確認，系統不自動更改主規則

## 股票觀察清單

系統每日分析 `rules/watchlist.md` 中定義的觀察標的。你可以隨時修改此檔案來增減觀察股票，系統會在下一次早盤預測時自動讀取。

典型格式：

| 代號 | 名稱 | 產業 | 當前部位 | 備註 |
|------|------|------|---------|------|
| 2330 | 台積電 | 半導體 | 無 / 持有 | 核心追蹤標的 |
| 2317 | 鴻海 | 電子代工 | 觀望 | 次要追蹤 |

## 工作流程

### 📊 早盤預測流程（每天 08:00 自動執行）

**步驟 1：3 個 Sub Agent 並行搜集資料**

1. **技術面 Agent**
   - 搜集：觀察清單中每檔股票的昨日 K 線、今日預開、相關指數走勢、主要均線、成交量
   - 分析方向：短期（5/20 日線）、中期（20/60 日線）趨勢方向；支撑壓力位；波段目標位
   - 輸出：每股技術方向（偏多/偏空/盤整）+ 關鍵支撑壓力位 + 波段目標位

2. **流動性 Agent**
   - 搜集：整體市場（三大法人、融資融券、外資期貨、VIX）+ 各股個別三大法人買賣狀況
   - 分析方向：市場整體資金面方向；各股主力進出訊號
   - 輸出：市場整體方向（偏多/偏空/中性）+ 各股主力動向評估

3. **消息面 Agent**
   - 搜集：美股大盤（S&P500、Nasdaq）、行業指數（如費半 SOX）、全球總經、VIX、匯率；各股相關產業消息、催化劑、風險事件
   - 分析方向：外部環境對整體市場 & 各股產業的影響
   - 輸出：整體消息方向（偏多/偏空/中性）+ 各股相關催化劑與風險

**步驟 2：主 Agent 整合與討論**

主 Agent 依次檢閱三個 Sub Agent 的報告，指出異議、共識點、矛盾之處，進行綜合判斷。
針對每檔觀察股票，綜合三方資訊輸出波段進場/持有/減碼/出場建議。

**步驟 3：輸出結論並記錄**

主 Agent 為每檔觀察股票匯總結論，包括：
- 波段走向（上漲/下跌/盤整）
- 波段建議（🟢 進場 / 🔵 加碼 / ⚪ 持有 / 🟡 減碼 / 🔴 出場 / ⏸️ 觀望）
- 目標價位（依支撑壓力位或技術型態）
- 停損位置
- 信心程度（高/中/低）
- 關鍵依據
- 主要風險
- 進場條件（若建議「觀望」，說明什麼條件才考慮進場）

寫入 `reports/YYYY-MM-DD.md`

### 📝 收盤檢討流程（每天 16:00 自動執行）

**步驟 1：讀取當日預測**

讀取 `reports/YYYY-MM-DD.md` 中的早盤預測結論。

**步驟 2：對照實際走勢**

針對每檔觀察股票記錄：
- 開盤價、收盤價、漲跌幅、漲跌點數
- 最高、最低
- 成交量
- 是否符合走向預測
- 是否達到目標位或觸發停損

**步驟 3：根本原因分析（若不符合）**

若預測與實際走勢不符，分析原因：
- 是否遇到黑天鵝事件？
- 是否有技術面的變數未預見？
- 流動性是否突變？
- 消息面有無新發展？

#### 📋 根本原因分析的信息完整性評估

**重要區分：預測失誤的真實原因**

根本原因可分為兩類，需要明確區分責任：

**A. 信息不足（事前無法預知）→ 不是系統缺陷**
- 例：「個股大漲但公布財報低於預期」（新信息）
- 理由：早盤 8:00 無法預知午盤公布的新財報結果
- 處理：記錄在報告供參考，但**不作為系統改進的依據**
- 標註：用 ⚠️ 表示「信息不足導致」

**B. 邏輯矛盾或風險未充分反映（事前應該預知）→ 這是系統缺陷** 
- 例：「識別了題材股高位風險，卻建議進場」
- 理由：早盤報告已有此資訊，應該反映在波段建議中
- 處理：納入「系統缺陷識別」，制定改進方案
- 標註：用 🔴 表示「系統邏輯缺陷」

**實操判斷標準**

檢查每個根本原因時，問自己：
1. 「這個信息在早盤 8:00 是否已知？」
   - YES → 如果沒反映在建議中，這是**系統缺陷**（🔴）
   - NO → 這是**信息不足**（⚠️），不需要改進系統

2. 「這個信息是否早盤預測已經識別但沒充分利用？」
   - YES → **系統缺陷**（🔴），需要改進邏輯
   - NO → **黑天鵝/信息不足**（⚠️），記錄供參考

---

**步驟 4：交易結果評估** ⭐ **重要**

若當日給出波段建議（🟢 進場 / 🔵 加碼 / ⚪ 持有 / 🟡 減碼 / 🔴 出場），應分析實際交易結果：

- **若用戶按建議操作，實際結果是賺還是虧？**
  - 若建議是「🟢 進場」或「🔵 加碼」，但隔日或數天內下跌且觸發停損 → **系統有缺陷**
  - 若建議是「⏸️ 觀望」，用戶自行操作且虧損 → 使用者自行承擔
  
- **波段持倉的浮盈浮虧趨勢是否符合預期？** → 記錄根本原因
  - 目標位達成情況
  - 停損是否被觸發
  - 趨勢是否反轉
  - 其他
  
- **是預測邏輯問題，還是波段建議定義問題？**
  - 若預測「方向對但幅度錯」+ 用戶浮虧 → 波段目標位設定有缺陷
  - 若預測「未充分警示不適合進場」+ 用戶虧損 → 系統評估邏輯不足
  - 記錄系統沒有顧慮到的因素到 learning_list.md

**步驟 5：規則驗證與優化**

根據步驟 3 和 4 分析出的虧損根本原因，評估是否值得學習：

- **識別虧損根本原因**
  - 來自步驟 3 的未預見變數
  - 來自步驟 4 的交易失敗原因
  
- **評估原因的學習價值**
  - 此原因是否具有重複性（會在未來再次出現）？
  - 學習此原因是否能有效提高勝率？
  - 是否可量化為交易規則？
  
- **高價值原因的處理**
  - 記錄到 `rules/learning_list.md`，標記為「待驗證 (1/10)」
  - 執行規則驗證（在後續交易日中測試該規則有效性）
  - 當規則驗證成功率達 70% 以上時，提出升級至主規則庫

- **系統層級的缺陷**
  - 若發現波段建議定義不清、禁止進場日條件缺失等系統評估邏輯問題
  - 詳細說明缺陷內容，詢問用戶是否更新 CLAUDE.md 相關定義
  - 用戶確認後，更新 CLAUDE.md，防止類似虧損再次發生

寫入 `reports/YYYY-MM-DD.md` 的檢討區段。

## Sub Agent 詳細分工

### 技術面分析 Agent

**搜集來源**（針對 watchlist 中每檔股票）
- Yahoo Finance / TradingView：股票昨日 OHLC、今日預開盤、成交量
- 相關指數走勢對比（如半導體股參考 TAIEX、0050 走勢）
- 短期均線（5MA、20MA、60MA）位置
- 月報成交量統計

**分析重點**
- 短期趨勢方向（5日線、20日線方向）
- 上升/下降通道的支撑壓力位
- 是否形成特殊K線形態（十字星、吞沒線等）
- 成交量是否放大或萎縮
- 波段目標位（基於上升通道頂部、前期高點等）

**輸出格式**
```
## 技術面分析 (Agent 1)

### [股票代號] [股票名稱]
- **趨勢方向**：偏多 / 偏空 / 中性
- **支撑位**：___
- **壓力位**：___
- **波段目標位**：___
- **成交量信號**：放大 / 萎縮 / 平均
- **形態判斷**：...
- **摘要**：今日應關注 ___ 位置的表現
```

### 流動性分析 Agent

**搜集來源**
- TWSE 三大法人：外資、投信、自營昨日買賣超（包含個股）
- TWSE 融資融券：融資增減、融券增減趨勢
- 期交所：台指期外資淨多空部位
- VIX 恐慌指數

**分析重點**
- 整體市場三大法人近期買賣趨勢（是否一致或分歧？）
- 各股個別主力（外資）的進出訊號
- 融資融券變化是否顯示市場信心
- 外資期貨淨部位（多/空傾向）

**輸出格式**
```
## 流動性分析 (Agent 2)

### 整體市場
- **資金方向**：偏多 / 偏空 / 中性
- **三大法人（昨日）**：外資 / 投信 / 自營 買賣超情況
- **融資融券**：融資增減 / 融券增減
- **期貨淨多空**：...

### 各股主力動向
| 代號 | 名稱 | 外資買賣超(近3日) | 評估 |
|------|------|-----------------|------|
| 2330 | 台積電 | 買超 | 主力看多 |
| ...  | ... | ... | ... |
```

### 消息面分析 Agent

**搜集來源**
- 美股昨晚收盤（S&P500、Nasdaq、Philadelphia Semiconductor Index/SOX）
- 全球總經指標（匯率 USD/TWD、油價、商品期貨）
- VIX 恐慌指數、波動率指標
- 台股今日重大消息、產業動態
- 突發事件風險（地緣政治、自然災害等）

**分析重點**
- 美股走勢對台股整體的帶動
- 產業指數（如 SOX）對相關個股的影響
- 是否有重大政策、經濟數據發布
- 是否有突發事件風險
- 全球風險情緒升溫或緩和

**輸出格式**
```
## 消息面分析 (Agent 3)

### 外部環境
- **美股表現（昨晚）**：
  - S&P500：___ (%漲跌)
  - Nasdaq：___ (%漲跌)
  - SOX（費半）：___ (%漲跌)
- **市場情緒**：樂觀 / 中性 / 謹慎 / 恐慌
- **VIX 指數**：___（漲/跌）
- **重大消息**：無 / 有（說明）
- **風險評估**：低 / 中 / 高

### 產業與個股催化劑
| 代號 | 相關產業消息 | 影響 |
|------|-----------|------|
| 2330 | COMPUTEX 準備、AI 供應鏈 | 正面 |
| ...  | ... | ... |

- **摘要**：全球背景對台股與各股的影響
```

### 主 Agent 整合任務

**責任**

1. **評估各 Agent 輸出品質**
   - 資料是否完整？
   - 邏輯推導是否合理？
   - 是否有明顯矛盾？

2. **協調與討論**
   - 若三個方向一致 → 信心提高
   - 若有分歧 → 分析哪一方更重要、哪一方可能失效的原因

3. **針對每檔觀察股票做出波段判斷**
   - 走向預測：上漲 / 下跌 / 盤整（未來數天到數週的中期趨勢）
   - 波段建議：🟢 進場 / 🔵 加碼 / ⚪ 持有 / 🟡 減碼 / 🔴 出場 / ⏸️ 觀望
   - 信心程度：高(3個一致) / 中(2個一致或矛盾但不嚴重) / 低(全分歧)

4. **識別關鍵因子**
   - 未來數天對該股最重要的看點是什麼？
   - 什麼情況下預測失效？

**輸出格式**
```markdown
## 📊 早盤預測結論

### 市場背景概述
[消息面 + 整體資金面摘要]

### 各股波段建議

#### [代號] [名稱]
- **波段走向**：上漲 / 下跌 / 盤整
- **波段建議**：🟢 進場 / 🔵 加碼 / ⚪ 持有 / 🟡 減碼 / 🔴 出場 / ⏸️ 觀望
- **目標價位**：___（依支撑壓力位或技術型態）
- **停損位置**：___
- **信心程度**：高 / 中 / 低
- **關鍵依據**：[最重要的 2-3 個因素]
- **主要風險**：[未來數天可能的意外情況]
- **進場條件**：（若建議「觀望」，說明什麼條件才考慮進場）
```

## 規則管理機制

### 規則的四種狀態

```
📍 發現新規則
   ↓
[待驗證] ← 新規則記錄到 learning_list.md，累計 0/10 次
   ↓ (驗證中)
[驗證中] ← 每日收盤更新驗證記錄
   ↓ (驗證達 10 次且成功 7 次)
[主規則] ← 等待用戶確認後加入 CLAUDE.md 主規則庫
   ↓ (主規則運行中)
[降級觀察] ← 連續失敗 3 次或近 15 次失敗 6 次，等待用戶確認後降級
```

### 升級條件（Learning List → 主規則）

**驗證標準**
- **樣本數**：累計 10 次觀察
- **成功率**：至少 7 次成功（70%）
- **用戶確認**：必須詢問用戶確認才能升級 ← **重要**

**流程**
1. 規則驗證滿 10 次且成功 ≥ 7 次
2. 系統標記為「準備升級」
3. 詢問用戶：「規則 #XXX 已驗證成功 7/10，是否升級為主規則？」
4. 用戶確認後，規則寫入 CLAUDE.md「主規則庫」

### 降級觀察條件（主規則出現異常）

**黃旗警告** ⚠️
- 主規則連續失敗 3 次
- 系統自動通知用戶，並進入「降級觀察期」
- 繼續累計驗證，但監控更頻繁

**降級提議** 🔴
- 主規則在近 15 次觀察中失敗 6 次（失敗率 40%）
- 系統詢問用戶：「規則 #XXX 失敗率達 40%，是否降級回學習清單？」
- 用戶確認後，規則降級回 learning_list.md

**特例處理**
- 若某次失敗有明顯特殊原因（黑天鵝、政策突變、限制措施等）
- 標記為「特例」，該次失敗不計入失敗次數
- 特例認定需要用戶確認

### learning_list.md 格式

```markdown
# 規則學習清單

## 規則 #001 - [規則名稱]

- **規則內容**：[簡明描述此規則的交易邏輯]
- **適用對象**：個股波段 / 市場整體環境評估
- **觸發條件**：[具體的市場條件，如「股價創新高後回檔」等]
- **波段建議**：🟢 進場 / 🔵 加碼 / ⚪ 持有 / 🟡 減碼 / 🔴 出場 / ⏸️ 觀望
- **發現日期**：YYYY-MM-DD
- **發現背景**：[在哪次檢討中發現此規則]

**驗證狀態**：待驗證 (0/10)

**驗證記錄**：

| 日期 | 條件是否觸發 | 建議 | 實際結果 | 符合 | 備註 |
|------|-------------|------|---------|------|------|
| YYYY-MM-DD | ✓ | 進場 | 上漲 | ✅ | - |
| YYYY-MM-DD | ✓ | 進場 | 下跌 | ❌ | 黑天鵝（特例）|
```

## 系統設定與排程

### 預測與檢討觸發

系統使用 `/schedule` 技能設定**遠端自動排程**（在雲端執行）：

1. **早盤預測排程**（每個交易日早上 8:00）
   - 讀取 `rules/watchlist.md` 中定義的觀察標的
   - 觸發 3 個 Sub Agent 蒐集資料
   - 主 Agent 進行綜合分析，針對每檔股票輸出波段建議
   - 將結論寫入 `reports/YYYY-MM-DD.md` 的預測段

2. **收盤檢討排程**（每個交易日 16:00）
   - 讀取當日預測結論
   - 對照實際走勢
   - 進行根本原因分析
   - 若有新規則，寫入 `rules/learning_list.md`
   - 將檢討記錄寫入 `reports/YYYY-MM-DD.md` 的檢討段

### 配置方式

完成系統建立後，使用 `/schedule` 技能啟動排程：

```
/schedule 早盤預測
描述：每天 08:00 台股開市前，調用3個Sub Agent搜集各股技術/流動性/消息面資料，進行波段預測並寫入 reports/YYYY-MM-DD.md
頻率：工作日早上 08:00 (台灣時間)

/schedule 收盤檢討
描述：每天 16:00 讀取當日預測，對照實際走勢進行檢討，若有新規則加入 rules/learning_list.md
頻率：工作日下午 16:00 (台灣時間)
```

**優勢**
- ✅ 在雲端執行，**不需要電腦開機或 Claude Code 執行**
- ✅ 預測和檢討結果自動推送到 github
- ✅ 可隨時查看執行結果

**注意事項**
- 非交易日（週末、假日、股市休市日）自動跳過
- 需要 github 遠端倉庫連接（結果寫入遠端）
- 使用 `/schedule list` 查看當前排程、`/schedule cancel <ID>` 暫停排程

## 系統使用提示

1. **我如何添加或移除觀察股票？**
   - 直接編輯 `rules/watchlist.md`，在「主動追蹤」區添加或移除股票即可
   - 系統會在下一次早盤預測時自動讀取新清單

2. **波段建議是否應該遵循？**
   - 波段建議是經過驗證的交易參考，但**非強制執行指令**
   - 最終決策權在用戶手中
   - 系統的目標是提供客觀的分析框架，而非承諾獲利

3. **波段建議失效了怎麼辦？**
   - 系統會監控波段建議表現，並在出現黃旗時通知
   - 用戶可隨時手動修改 watchlist，降級或移除失效的股票
   - 降級後的規則會重新進入學習清單

4. **我想手動添加一個規則怎麼做？**
   - 直接編輯 `rules/learning_list.md`，添加新規則
   - 標記為「待驗證 (0/10)」
   - 系統會在後續檢討中驗證此規則

5. **我想暫停排程怎麼做？**
   - 使用 `/schedule list` 查看當前排程
   - 使用 `/schedule cancel <排程ID>` 暫停排程
   - 需要時可重新啟動排程
