# 00631L 正2當沖 AI 自動化系統

## 系統說明

這是一套專為台股 **00631L（元大台灣50正2）** 設計的日內交易（當沖）輔助系統。

每天早上 8:00 前，系統自動調用 3 個 Sub Agent 進行並行資料蒐集，進行綜合分析後輸出預測。
收盤後進行檢討，並透過規則驗證機制逐步優化交易規則庫。

### 核心理念
- **自動化**：每日預測與檢討由排程自動執行，無需手動觸發
- **系統化**：每個決策都有明確的依據（技術面、流動性、消息面）
- **自我優化**：規則透過持續驗證逐步升級或調整
- **用戶控制**：任何規則變更都需要用戶確認，系統不自動更改主規則

## 工作流程

### 📊 早盤預測流程（每天 08:00 自動執行）

**步驟 1：3 個 Sub Agent 並行搜集資料**

1. **技術面 Agent**
   - 搜集：00631L 昨日K線、今日預開、台灣50指數走勢、主要均線、成交量
   - 輸出：技術面方向（偏多/偏空/中性）+ 關鍵支撑壓力位

2. **流動性 Agent**
   - 搜集：三大法人昨日買賣超、融資融券變化、外資期貨淨多空
   - 輸出：資金面方向（偏多/偏空/中性）+ 主力方向判斷

3. **消息面 Agent**
   - 搜集：美股昨晚收盤（S&P500、Nasdaq、費半）、台股今日重大消息、VIX恐慌指數、特殊風險事件
   - 輸出：消息面方向（偏多/偏空/中性）+ 風險提示

**步驟 2：主 Agent 整合與討論**

主 Agent 依次檢閱三個 Sub Agent 的報告，指出異議、共識點、矛盾之處，進行綜合判斷。

**步驟 3：輸出結論並記錄**

主 Agent 匯總結論，包括：
- 走向預測（上漲/下跌/盤整）
- 當沖建議（✅ 適合 / ⚠️ 謹慎 / ❌ 不適合）
- 信心程度（高/中/低）
- 關鍵依據
- 主要風險

寫入 `reports/YYYY-MM-DD.md`

---

### 📝 收盤檢討流程（每天 15:00 自動執行）

**步驟 1：讀取當日預測**

讀取 `reports/YYYY-MM-DD.md` 中的早盤預測結論。

**步驟 2：對照實際走勢**

- 開盤價、收盤價、漲跌幅
- 最高、最低
- 成交量
- 是否符合走向預測
- 是否符合當沖建議

**步驟 3：根本原因分析（若不符合）**

若預測與實際走勢不符，分析原因：
- 是否遇到黑天鵝事件？
- 是否有技術面的變數未預見？
- 流動性是否突變？
- 消息面有無新發展？

**步驟 4：交易結果評估** ⭐ **重要**

若當日給出當沖建議（✅ 適合 / ⚠️ 謹慎 / ❌ 不適合），應分析實際交易結果：

- **若用戶按建議操作，實際結果是賺還是虧？**
  - 若建議是「✅ 適合」或「⚠️ 謹慎」，但實際虧損 → **系統有缺陷**
  - 若建議是「❌ 不適合」，用戶仍操作且虧損 → 使用者自行承擔
  
- **虧損發生在哪個環節？** → 記錄根本原因
  - 破止損（支撑位失效）
  - 波動被磨（日內高波動導致出場）
  - 漲幅不足（預期過樂觀）
  - 其他
  
- **是預測邏輯問題，還是系統建議定義問題？**
  - 若預測「方向對但幅度錯」+ 用戶虧損 → 當沖建議定義有缺陷
  - 若預測「未充分警示不適合」+ 用戶虧損 → 系統評估邏輯不足
  - 記錄系統沒有顧慮到的因素到 learning_list.md

**步驟 5：規則驗證與優化**

根據步驟 3 和 4 分析出的虧損根本原因，評估是否值得學習：

- **識別虧損根本原因**
  - 來自步驟 3 的未預見變數
  - 來自步驟 4 的交易失敗原因
  
- **評估原因的學習價值**
  - 此原因是否具有重複性（會在未來再次出現）？
  - 學習此原因是否能有效提高勝率？
  - 是否可量化為交易規則？
  
- **高價值原因的處理**
  - 記錄到 `rules/learning_list.md`，標記為「待驗證 (1/10)」
  - 執行規則驗證（在後續交易日中測試該規則有效性）
  - 當規則驗證成功率達 70% 以上時，提出升級至主規則庫

- **系統層級的缺陷**
  - 若發現當沖建議定義不清、禁止當沖日條件缺失等系統評估邏輯問題
  - 詳細說明缺陷內容，詢問用戶是否更新 CLAUDE.md 相關定義
  - 用戶確認後，更新 CLAUDE.md，防止類似虧損再次發生

寫入 `reports/YYYY-MM-DD.md` 的檢討區段。

## Sub Agent 詳細分工

### 技術面分析 Agent

**搜集來源**
- Yahoo Finance：00631L 昨日 OHLC、今日預開盤、成交量
- Yahoo Finance：0050（台灣50指數）走勢對比
- TradingView：短期均線（5MA、20MA、60MA）位置
- TWSE：月報成交量統計

**分析重點**
- 短期趨勢方向（5日線、20日線方向）
- 上升/下降通道的支撑壓力位
- 昨日是否形成特殊K線形態（十字星、吞沒線等）
- 成交量是否放大或萎縮

**輸出格式**
```
## 技術面分析 (Agent 1)
- **趨勢方向**：偏多 / 偏空 / 中性
- **關鍵位置**：
  - 支撑位：___
  - 壓力位：___
- **成交量信號**：放大 / 萎縮 / 平均
- **形態判斷**：...
- **摘要**：今日開盤應關注 ___ 位置的表現
```

### 流動性分析 Agent

**搜集來源**
- TWSE 三大法人：`twse.com.tw/fund/T86` → 昨日外資、投信、自營淨買賣
- TWSE 融資融券：融資增減、融券增減趨勢
- 期交所：台指期外資淨多空部位

**分析重點**
- 三大法人的近期買賣趨勢（是否一致或分歧？）
- 融資融券變化是否顯示主力進出
- 外資期貨淨部位（多/空傾向）

**輸出格式**
```
## 流動性分析 (Agent 2)
- **資金方向**：偏多 / 偏空 / 中性
- **三大法人**：
  - 外資：___ (近5日趨勢)
  - 投信：___ (近5日趨勢)
  - 自營：___ (近5日趨勢)
- **融資融券**：融資增減 / 融券增減
- **期貨淨多空**：...
- **摘要**：主力態度傾向 ___
```

### 消息面分析 Agent

**搜集來源**
- WebSearch：美股昨晚收盤（S&P500、Nasdaq、費城半導體指數）
- WebSearch：台股今日重大消息、個股公告
- WebSearch/CBOE：VIX恐慌指數、波動率指標
- WebSearch：匯率（USD/TWD）、油價、商品期貨

**分析重點**
- 美股走勢對台股的可能帶動
- 是否有重大政策、經濟數據發布
- 是否有突發事件風險（地緣政治、自然災害等）
- 全球風險情緒是否升溫或緩和

**輸出格式**
```
## 消息面分析 (Agent 3)
- **美股表現**：
  - S&P500：___ (%漲跌)
  - Nasdaq：___ (%漲跌)
  - 費半：___ (%漲跌)
- **市場情緒**：樂觀 / 中性 / 謹慎 / 恐慌
- **VIX 指數**：___（漲/跌）
- **重大消息**：無 / 有（說明）
- **風險評估**：低 / 中 / 高
- **摘要**：全球背景對台股影響 ___
```

### 主 Agent 整合任務

**責任**

1. **評估各 Agent 輸出品質**
   - 資料是否完整？
   - 邏輯推導是否合理？
   - 是否有明顯矛盾？

2. **協調與討論**
   - 若三個方向一致 → 信心提高
   - 若有分歧 → 分析哪一方更重要、哪一方可能失效的原因

3. **做出綜合判斷**
   - 走向預測：上漲 / 下跌 / 盤整
   - 當沖適合度：✅ 適合 / ⚠️ 謹慎 / ❌ 不適合
   - 信心程度：高(3個一致) / 中(2個一致) / 低(全分歧)

4. **識別關鍵因子**
   - 今日交易最重要的看點是什麼？
   - 什麼情況下預測失效？

**輸出格式**
```markdown
## 📊 早盤預測結論

### 三方綜合判斷
[說明三個Agent的觀點，指出共識與分歧]

### 主 Agent 結論
- **走向預測**：上漲 / 下跌 / 盤整
- **當沖建議**：✅ 適合 / ⚠️ 謹慎 / ❌ 不適合
- **信心程度**：高 / 中 / 低
- **關鍵依據**：[最重要的 2-3 個因素]
- **主要風險**：[今日可能的意外情況]
- **交易關注點**：[具體的支撑/壓力位、突破位等]
```

## 規則管理機制

### 規則的四種狀態

```
📍 發現新規則
   ↓
[待驗證] ← 新規則記錄到 learning_list.md，累計 0/10 次
   ↓ (驗證中)
[驗證中] ← 每日收盤更新驗證記錄
   ↓ (驗證達 10 次且成功 7 次)
[主規則] ← 等待用戶確認後加入 CLAUDE.md 主規則庫
   ↓ (主規則運行中)
[降級觀察] ← 連續失敗 3 次或近 15 次失敗 6 次，等待用戶確認後降級
```

### 升級條件（Learning List → 主規則）

**驗證標準**
- **樣本數**：累計 10 次觀察
- **成功率**：至少 7 次成功（70%）
- **用戶確認**：必須詢問用戶確認才能升級 ← **重要**

**流程**
1. 規則驗證滿 10 次且成功 ≥ 7 次
2. 系統標記為「準備升級」
3. 詢問用戶：「規則 #XXX 已驗證成功 7/10，是否升級為主規則？」
4. 用戶確認後，規則寫入 CLAUDE.md「主規則庫」

### 降級觀察條件（主規則出現異常）

**黃旗警告** ⚠️
- 主規則連續失敗 3 次
- 系統自動通知用戶，並進入「降級觀察期」
- 繼續累計驗證，但監控更頻繁

**降級提議** 🔴
- 主規則在近 15 次觀察中失敗 6 次（失敗率 40%）
- 系統詢問用戶：「規則 #XXX 失敗率達 40%，是否降級回學習清單？」
- 用戶確認後，規則降級回 learning_list.md

**特例處理**
- 若某次失敗有明顯特殊原因（黑天鵝、政策突變、限制措施等）
- 標記為「特例」，該次失敗不計入失敗次數
- 特例認定需要用戶確認

### learning_list.md 格式

```markdown
# 規則學習清單

## 規則 #001 - [規則名稱]

- **規則內容**：[簡明描述此規則的交易邏輯]
- **觸發條件**：[具體的市場條件，如「美股漲 >1%」「VIX < 12」等]
- **交易方向**：做多 / 做空 / 不交易
- **發現日期**：YYYY-MM-DD
- **發現背景**：[在哪次檢討中發現此規則]

**驗證狀態**：待驗證 (0/10)

**驗證記錄**：

| 日期 | 條件是否觸發 | 預測方向 | 實際結果 | 符合 | 備註 |
|------|-------------|---------|---------|------|------|
| YYYY-MM-DD | ✓ | 上漲 | 上漲 | ✅ | - |
| YYYY-MM-DD | ✓ | 下跌 | 上漲 | ❌ | 黑天鵝（特例）|

---
```

## 系統設定與排程

### 預測與檢討觸發

系統使用 `/schedule` 技能設定**遠端自動排程**（在雲端執行）：

1. **早盤預測排程**（每個交易日早上 8:00）
   - 觸發 3 個 Sub Agent 蒐集資料
   - 主 Agent 進行綜合分析
   - 將結論寫入 `reports/YYYY-MM-DD.md` 的預測段

2. **收盤檢討排程**（每個交易日 15:00）
   - 讀取當日預測結論
   - 對照實際走勢
   - 進行根本原因分析
   - 若有新規則，寫入 `rules/learning_list.md`
   - 將檢討記錄寫入 `reports/YYYY-MM-DD.md` 的檢討段

### 配置方式

完成系統建立後，使用 `/schedule` 技能啟動排程：

```
/schedule 早盤預測
描述：每天 08:00 台股開市前，調用3個Sub Agent搜集技術/流動性/消息面資料，進行預測並寫入 reports/YYYY-MM-DD.md
頻率：工作日早上 08:00 (台灣時間)

/schedule 收盤檢討
描述：每天 15:00 讀取當日預測，對照實際走勢進行檢討，若有新規則加入 rules/learning_list.md
頻率：工作日下午 15:00 (台灣時間)
```

**優勢**
- ✅ 在雲端執行，**不需要電腦開機或 Claude Code 執行**
- ✅ 預測和檢討結果自動推送到 github
- ✅ 可隨時查看執行結果

**注意事項**
- 非交易日（週末、假日、股市休市日）自動跳過
- 需要 github 遠端倉庫連接（結果寫入遠端）
- 使用 `/schedule list` 查看當前排程、`/schedule cancel <ID>` 暫停排程

## 系統使用提示

1. **我是否應該遵循主規則進行交易？**
   - 主規則是經過驗證的交易參考，但**非強制執行指令**
   - 最終決策權在用戶手中
   - 系統的目標是提供客觀的分析框架，而非承諾獲利

2. **主規則失效了怎麼辦？**
   - 系統會監控主規則表現，並在出現黃旗時通知
   - 用戶可隨時手動降級任何規則
   - 降級後的規則會重新進入學習清單

3. **我想手動添加一個規則怎麼做？**
   - 直接編輯 `rules/learning_list.md`，添加新規則
   - 標記為「待驗證 (0/10)」
   - 系統會在後續檢討中驗證此規則

4. **我想暫停排程怎麼做？**
   - 使用 `/schedule list` 查看當前排程
   - 使用 `/schedule cancel <排程ID>` 暫停排程
   - 需要時可重新啟動排程
